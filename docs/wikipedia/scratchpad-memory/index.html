Scratchpad memory

14 min read

Based on Wikipedia: Scratchpad memory

In 1975, the Fairchild F8 microprocessor, a chip that would help define the dawn of the personal computing era, contained a mere 64 bytes of a specific type of internal memory. It was not the main memory where programs lived, nor was it the registers where the processor performed its immediate math. It was something smaller, faster, and entirely under the direct control of the programmer: scratchpad memory. This tiny, 64-byte workspace was the precursor to a fundamental architectural philosophy that would eventually power the supercomputers, game consoles, and AI accelerators of the 21st century. While modern desktop users have spent decades relying on the invisible hand of hardware-managed caches to make their systems feel fast, a different breed of computing hardware has always operated on a more explicit, transparent, and often more efficient principle: the scratchpad.

To understand the significance of this memory architecture, one must first strip away the abstraction of the modern "black box" computer. In a typical consumer processor found in a laptop or a smartphone today, data movement is largely a mystery to the software. The hardware automatically decides what to keep in the fast L1 cache, what to evict to the L2 or L3, and when to fetch new data from the slow main memory. This system, known as a cache, relies on "spatial and temporal locality" assumptions—the belief that if you use a piece of data once, you will likely use it again soon, or that you will need the data next to it. For general-purpose computing running legacy software, this is a miracle of engineering. It allows a program written in 1990 to run on a 2026 processor without modification, even as the underlying memory hierarchy changes drastically.

But there is a cost to this convenience. The hardware cache controller consumes a significant amount of silicon real estate and power to monitor data, predict access patterns, and maintain "cache coherence" across multiple cores. When the hardware guesses wrong—a "cache miss"—the processor stalls, waiting for data to trickle in from main memory. In systems where timing is critical, or where the data access patterns are too complex for simple hardware prediction, this unpredictability is a fatal flaw.

Enter the scratchpad. Also known as scratchpad RAM, local store, or bump storage, this is an internal, high-speed memory used for the temporary storage of calculations, data, and work in progress. The name is not merely a metaphor; it is a direct functional analogy. Just as a writer uses a physical pad of paper to sketch preliminary notes, perform quick calculations, or draft a section before committing it to the final manuscript, a processor uses scratchpad memory to hold small items of data for rapid retrieval without the overhead of a complex caching hierarchy.

The fundamental difference lies in control. In a system with scratchpads, the memory is not managed by the hardware's automatic logic. It is explicitly manipulated by the application. The software developer must issue specific instructions, often utilizing Direct Memory Access (DMA) engines, to move data from the main memory into the scratchpad, perform the necessary calculations, and then move the results back. This shifts the burden of optimization from the silicon to the software engineer. It sounds like a step backward, a regression to the assembly-language days of yore, but it offers a profound advantage: determinism.

When you know exactly where your data is, you know exactly how long it will take to access it. There are no cache misses. There are no unpredictable stalls. The latency is fixed and known. This is why scratchpads are the heartbeat of real-time systems and embedded applications where a missed deadline can mean a crashed car, a failed medical procedure, or a dropped video frame. In these environments, the system is often a Multiprocessor System-on-Chip (MPSoC), where dozens or even hundreds of specialized cores work in concert, each with its own private workspace.

The architecture of these systems creates a Non-Uniform Memory Access (NUMA) environment. Unlike a standard computer where every core sees the same memory latency, in a scratchpad-based system, the time it takes to access data varies wildly depending on which memory bank you are trying to reach. Accessing your own local scratchpad is instantaneous. Accessing another core's scratchpad requires a network transfer. Accessing the main memory is a slow, distant journey. This variability is not a bug; it is a feature that allows architects to tailor the system for massive parallelism without the "coherence tax" of keeping hundreds of caches in sync.

The history of scratchpad usage is a tour through the most ambitious and specialized computing projects of the last half-century. The TI-99/4A, a home computer from the early 1980s, featured 256 bytes of scratchpad memory on its 16-bit bus, which was large enough to hold the processor registers of the TMS9900 CPU. This allowed for incredibly fast context switching and interrupt handling. Decades later, the gaming industry would embrace this concept with a fervor that consumer PCs never matched.

The Sony PlayStation 1, released in 1994, famously replaced the traditional L1 cache with a 2 KB scratchpad in its R3000 CPU. This was a radical move for a consumer device. By placing the CPU stack directly in this scratchpad, the system guaranteed that the most critical data—the call stack and registers—was always available at the speed of light. Similarly, the Sega SuperH processors, used in various arcade and console hardware, allowed developers to lock cache lines to an address outside of main memory, effectively turning a portion of the cache into a user-controlled scratchpad.

However, the true evolution of the scratchpad arrived with the rise of the "many-core" future, a domain where the traditional cache hierarchy simply could not scale. The most iconic example of this is the Cell Broadband Engine, the custom processor designed for the Sony PlayStation 3. The Cell processor contained one Power Processing Element (PPE) and eight Synergistic Processing Elements (SPEs). The PPE was a general-purpose core, but the SPEs were the workhorses, and they operated entirely within their own 256 KB "local store."

This local store was a scratchpad in its purest form. It contained neither a cache nor a direct path to main memory. The SPEs could not "see" the main memory. To get data, the programmer had to program the SPE's DMA engine to pull data from the main memory into the local store. Once the calculation was done, the DMA engine pushed the results back. This architecture eliminated the need for complex hardware logic to check and update coherence between multiple caches. The design operated on a simple, powerful assumption: each processor's workspace is separate and private.

The benefits of this design were staggering. As the number of processors scaled, the power and area savings from removing the cache coherence hardware became the deciding factor in performance. The Cell processor demonstrated that for highly parallel, predictable workloads, explicit memory management could outperform even the most sophisticated hardware caches. However, this power came with a price. The data and instructions for applications running on the SPEs had to be managed entirely by software. If a task was too large to fit into the 256 KB local store, the programmer had to break it down into chunks and orchestrate the DMA transfers manually. It was a programming model that demanded deep understanding and discipline, but it delivered raw, unadulterated throughput.

The legacy of the Cell architecture rippled through the industry, influencing the design of GPUs and AI accelerators. NVIDIA, the giant of graphics processing, adopted the scratchpad philosophy for its CUDA architecture. In the GeForce 8800 series, released in 2006, and subsequent generations like the Fermi architecture (GeForce 400 series), NVIDIA introduced "Shared Memory." This is, in essence, a scratchpad. It provides 16 KB of high-speed, software-managed memory per thread block. When running General-Purpose GPU (GPGPU) tasks, developers can use this Shared Memory to stage data, avoiding the latency of the global memory and the unpredictability of the texture cache. It allows for fine-tuned optimization that is impossible with a standard cache hierarchy.

The philosophy extended beyond graphics. The Ageia PhysX chip, a dedicated physics processing unit, utilized scratchpad RAM in a manner similar to the Cell. The theory was that for physics and collision calculations, a traditional cache hierarchy was less useful than software-managed data flow. By banking the memory and using a switch to manage transfers, the chip could process complex simulations with deterministic timing. Even Intel, the king of the general-purpose x86 market, began to experiment with this hybrid approach. The Knights Landing processor, a many-core architecture based on the Xeon Phi line, featured 16 GB of MCDRAM (Multi-Channel DRAM). This memory was unique in that it could be configured in software as a standard cache, as a scratchpad, or as a hybrid of the two. This gave developers the flexibility to choose the memory model that best suited their specific workload.

The modern AI revolution has further cemented the importance of scratchpad memories. Graphcore, a company designing AI accelerators, has built its entire architecture around scratchpad memories. Their Intelligence Processing Units (IPUs) utilize a massive, multiported shared scratchpad that allows thousands of cores to communicate and share data without the bottlenecks of traditional memory hierarchies. Similarly, the Movidius Myriad 2 vision processing unit, used in drones and autonomous vehicles, organizes its multicore architecture around a large shared scratchpad, enabling real-time computer vision processing with minimal latency.

It is worth noting that not all processors require a dedicated physical scratchpad to achieve these benefits. Some architectures, such as PowerPC, attempt to avoid the need for dedicated scratchpads or cacheline locking through the use of advanced cache control instructions. By marking an area of memory with "Data Cache Block: Zero" (allocating a line but setting its contents to zero instead of loading from main memory) and then discarding it with "Data Cache Block: Invalidate," the cache can be made to behave like a scratchpad. This maintains the generality of the hardware, allowing it to function correctly regardless of the actual cache size, while giving the programmer the hints needed to optimize performance. However, this is often a compromise, a way to simulate scratchpad behavior on hardware that wasn't designed for it.

The distinction between a cache and a scratchpad is not just technical; it is philosophical. A cache is a system built on hope—hope that the hardware will guess your next move correctly. A scratchpad is a system built on certainty. It is a tool for those who know exactly what they are doing and refuse to leave their performance to chance. This is why scratchpads are rarely found in mainstream desktop processors. The desktop market demands generality; it must run legacy software from generation to generation, and the available on-chip memory size may change unpredictably. A system that relies on explicit memory management would break every time a new processor generation arrived with a different memory layout.

But in the specialized worlds of embedded systems, game consoles, and high-performance computing, the scratchpad reigns supreme. The Cyrix 6x86 remains the only x86-compatible desktop processor to incorporate a dedicated scratchpad, a testament to how niche this approach is in the general market. In contrast, the SuperH processors in Sega consoles, the R3000 in the PS1, and the Cell in the PS3 all leveraged scratchpads to push the boundaries of what was possible with the silicon of their time.

The future of computing points toward an even greater reliance on these explicit memory models. As we move into the era of "many-core" processors, with hundreds or thousands of cores on a single chip, the overhead of maintaining cache coherence becomes unsustainable. The design of the Cell processor, where each processor's workspace is separate and private, is becoming the blueprint for the future. The elimination of hardware coherence logic allows for massive scaling, provided the software can manage the data flow.

This shift requires a new kind of programmer. It demands a return to the roots of computer science, where the programmer is the architect of the data flow, not just the author of the algorithm. The ability to utilize strided DMA to access rectangular data sets, or to chain DMA transfers between local stores in a network-on-a-chip, opens up possibilities that are invisible to the cache-based programmer. For example, in pipeline processing, one can achieve the effect of increasing the L1 cache size by splitting a job into smaller chunks and moving them explicitly between scratchpads. In merge sort algorithms, the working set can be extended to fit within the aggregate scratchpad memory of multiple cores, creating a "sweet spot" for performance that bypasses the main memory entirely.

The interplay between these scratchpads also creates a unique form of interprocessor communication. In the Cell architecture, the ability to perform DMA between local stores meant that the local store of one processor could serve as the shared memory for another. This dual role—acting as both a private workspace and a shared communication buffer—was a brilliant design choice. It meant that the other local stores were on a similar footing, viewed from one processor as the shared L2 cache in a conventional chip, but without the latency and coherence overhead of a true shared cache. The tradeoff was programming complexity and the need for explicit synchronization, but for high-performance applications, this was a price worth paying.

Even in the realm of 3D acceleration, where modern GPUs have largely moved to a stream-based approach with more in common with CPU caches, the scratchpad remains a vital tool. Digital Signal Processors (DSPs), which have long been used for vertex transformations in 3D accelerators, almost universally use scratchpad memory. The difference lies in the control; the DSP programmer directs the data flow, ensuring that the critical vertex data is always in the fastest possible memory when needed.

As we look toward the "inference kingdom" of AI, the scratchpad is not just a relic of the past; it is the foundation of the future. The ability to guarantee a unit can work without main memory contention is essential for the massive parallelism required by modern neural networks. Whether it is the MCDRAM in Intel's many-core processors, the shared memory in NVIDIA's GPUs, or the local stores in Graphcore's IPUs, the principle remains the same: explicit control over data movement yields superior performance and predictability.

The scratchpad memory is a reminder that in computing, sometimes the most advanced technology is not the most complex. It is the simplest, most direct path to the data. It is the rejection of the "black box" in favor of transparency and control. From the 64 bytes of the Fairchild F8 to the multi-gigabyte scratchpads of modern AI accelerators, this architectural choice has consistently provided the edge needed to break performance barriers. It is the pad of paper upon which the most complex calculations of our digital age are sketched, refined, and executed with precision.

In a world increasingly dominated by the unpredictability of automated systems, the scratchpad stands as a monument to human agency in computing. It is the place where the programmer's intent is executed without interference, where the latency is known, and where the performance is guaranteed. As we scale into the many-core future, the lessons of the scratchpad will become ever more critical. The hardware will continue to grow in complexity and scale, but the need for explicit, deterministic control over data will only grow. The scratchpad is not just a memory type; it is a philosophy of efficiency, a commitment to the principle that if you want to know how fast your system can go, you must build the road yourself.
